## 前言

*"道可道，非常道；名可名，非常名。"（《道德经》第一章）*

2026年2月，一个崭新的工程概念在AI领域迅速兴起——Harness Engineering（驾驭工程）。OpenAI团队发布了一份震撼业界的报告：他们在5个月内，以零行手写代码的方式，构建并交付了一个拥有约100万行代码的软件产品。所有代码——应用逻辑、测试、CI配置、文档、可观测性、内部工具——全部由Codex生成。

这一实验揭示了一个深刻的转变：软件工程师的核心工作正在从"写代码"转向"设计环境、指定意图、构建反馈循环"。这正是Harness Engineering的核心理念——"人类掌舵，Agent执行"（Humans steer. Agents execute.）。

Harness Engineering是继Prompt Engineering和Context Engineering之后，第三个核心AI工程概念。它的核心思想是：系统性地设计约束、反馈循环和持续改进机制，使AI Agent能够在大规模、长周期的软件开发中可靠运行。

本书尝试将《道德经》的东方智慧与Harness Engineering的西方技术相融合。我们将从"道法自然"理解Harness的本质——不是对抗Agent的不确定性，而是顺应其本性设计约束；从"无为而治"领悟声明式编排的优雅——人类定义意图，Agent自动执行；从"上善若水"学习适配与流动的设计哲学——Harness如流水般适应各种场景。

书中涵盖Harness的三大支柱：上下文工程、架构约束、熵管理。我们将深入解读OpenAI Codex Harness、Anthropic长周期Agent方案、Martin Fowler/Thoughtworks的分析，以及GitHub上2026年2月以来涌现的顶级开源项目——OpenClaw（33.5万Star）、DeerFlow 2.0（5.7万Star）、agency-agents（6万+Star）等。

每一章都将道德经原文与前沿技术深度结合，呈现公式推导、架构设计、代码示例、企业实践案例，力求做到理论与实践并重。愿这本书能为AI工程师们提供一种新的思考维度，在技术实现的同时，也能感受到东方哲学的深邃与优雅。

*"反者道之动，弱者道之用。"在Harness Engineering的世界里，约束不是限制，而是引导；反馈不是批评，而是进化；Harness不是禁锢，而是让Agent在正确道路上奔跑的完整装备。*

# 第一部分：Harness Engineering 基础认知

## 第一章 道法自然：Harness Engineering的本质认知

*"人法地，地法天，天法道，道法自然。"（《道德经》第二十五章）*

老子用这十六个字，揭示了宇宙万物运行的根本规律——"道法自然"。"自然"并非指自然界，而是指"自己如此"、"本来如此"的状态。道遵循的不是外在的强制规则，而是事物内在的本性和规律。这一思想对于理解Harness Engineering具有深刻的启示意义。

### 1.1 什么是Harness Engineering

Harness Engineering（驾驭工程）是继Prompt Engineering和Context Engineering之后，于2026年2月迅速兴起的第三个核心AI工程概念。它的核心思想是：系统性地设计约束、反馈循环和持续改进机制，使AI Agent能够在大规模、长周期的软件开发中可靠运行。

用一个经典比喻来理解：模型是引擎，Harness是整辆车。再强大的引擎，没有方向盘和刹车，哪儿也去不了。模型负责生成响应，Harness负责其他所有事情——人类审批、子Agent协调、文件系统访问、生命周期钩子、规划与执行。

或者用更古典的比喻："Harness"的本义是马具——缰绳、马鞍、嚼子——一整套将马匹力量导向正确方向、防止失控、保障长距离稳定运行的装备。LLM Agent就是那匹强大但不可预测的马，Harness就是让它在正确道路上奔跑的完整装备。

### 1.2 Harness Engineering的诞生

Harness Engineering的概念碎片化地存在于2025年末，但在2026年2月的短短几周内完成了"结晶"。以下是关键时间节点：

| 时间       | 事件                                                         | 人物/组织                                        |
| ---------- | ------------------------------------------------------------ | ------------------------------------------------ |
| 2026.02.05 | 发表博文'My AI Adoption Journey'，在第五步明确提出'Engineer the Harness' | Mitchell Hashimoto (HashiCorp联合创始人)         |
| 2026.02.11 | 发表'Harness engineering: leveraging Codex in an agent-first world'，详述百万行零手写代码实验 | OpenAI (Ryan Lopopolo)                           |
| 2026.02.17 | 发表分析文章，将OpenAI的Harness归纳为三大领域                | Martin Fowler / Birgitta Böckeler (Thoughtworks) |
| 2026.02    | 围绕'Models, Apps, and Harnesses'重组AI指南框架              | Ethan Mollick                                    |
| 2025.11.26 | 发表'Effective harnesses for long-running agents'（概念先驱） | Anthropic                                        |

### 1.3 OpenAI百万行代码实验

2026年2月11日，OpenAI发布了迄今最详尽的Harness Engineering实战报告。以下是实验的关键数据：

| 指标     | 数值                                  |
| -------- | ------------------------------------- |
| 起点     | 2025年8月，从一个空的Git仓库开始      |
| 约束     | 零行手写代码——所有代码均由Codex生成   |
| 团队规模 | 3名工程师（后扩展至7名）              |
| 代码量   | 约100万行                             |
| PR数量   | 约1,500个PR被合并                     |
| 吞吐量   | 平均每个工程师每天3.5个PR             |
| 效率估算 | 约为手工编码的10倍速度                |
| 产品状态 | 拥有内部每日活跃用户和外部Alpha测试者 |

### 1.4 Harness Engineering的五大原则

OpenAI团队从实验中提炼出五大Harness原则：

**原则一：设计环境，而非编写代码。** "当Agent遇到困难时，修复方法几乎从来不是'更努力'。人类工程师总是退一步问：'缺少什么能力？如何让它对Agent既可理解又可执行？'"

**原则二：机械化执行架构约束。** 定义了每个业务领域的依赖方向：Types → Config → Repo → Service → Runtime → UI。使用自定义Linters和结构测试自动检测违规。写在文档里不够——如果不能机械化执行，Agent就会偏离。

**原则三：仓库即唯一真相来源。** Slack讨论和Google Docs中的知识对Agent来说等于不存在。所有团队知识以版本控制的产物形式存放在仓库内。

**原则四：将可观测性接入Agent。** 将Chrome DevTools Protocol集成到Agent运行时，使Codex能捕获DOM快照和截图。暴露日志、指标、链路追踪，通过LogQL/PromQL/TraceQL查询。单次Codex运行可持续工作长达6小时以上。

**原则五：对抗熵增。** 最初，团队每周五花20%的时间手动清理"AI垃圾"（slop），这种方式无法规模化。解决方案：将"黄金原则"编码到仓库中，建立定期运行的后台Codex清理任务，自动扫描偏差、更新质量评分、提交定向重构PR。这类似于垃圾回收（GC）的持续技术债清理。

### 1.5 核心理念：人类掌舵，Agent执行

Harness Engineering的核心理念可以用一句话概括："Humans steer. Agents execute."（人类掌舵，Agent执行。）

这意味着：
- 人类的工作：设计环境、指定意图、构建反馈循环
- Agent的工作：执行具体任务、生成代码、运行测试
- Harness的作用：让人类能够高效地指导Agent完成有用的工作

这与道德经"无为而治"的思想高度契合——"我无为，而民自化；我好静，而民自正"。人类工程师通过设计Harness（环境、约束、反馈），让Agent能够自主运行，而不是事无巨细地控制每一步。

### 1.6 本章小结

本章从"道法自然"的哲学思想出发，介绍了Harness Engineering的本质——不是对抗Agent的不确定性，而是顺应其本性设计约束和反馈机制。我们学习了OpenAI百万行代码实验的五大原则，理解了"人类掌舵，Agent执行"的核心理念。

"知其雄，守其雌，为天下溪。"了解Agent的强大能力（雄），同时守护其需要约束的本性（雌），这是Harness工程师应有的智慧。

## 第二章 无为而治：从Prompt到Harness的演进

*"道常无为而无不为。侯王若能守之，万物将自化。"（《道德经》第三十七章）*

"无为"是道德经最核心的概念之一，也是最容易被误解的概念。"无为"并非什么都不做，而是"不妄为"、"不强为"，顺应事物的本性而行动。在AI工程领域，这一思想体现为从Prompt Engineering到Context Engineering再到Harness Engineering的演进——人类从直接控制输出，到设计环境让Agent自主运行。

### 2.1 AI工程的三次范式转移

AI工程经历了三次重要的范式转移，每一次都体现了"无为而治"的深化：

| 阶段                | 时间      | 核心问题         | 人类工作                 | Agent工作        |
| ------------------- | --------- | ---------------- | ------------------------ | ---------------- |
| Prompt Engineering  | 2022-2023 | 如何写出好提示词 | 精心编写提示词           | 根据提示词生成   |
| Context Engineering | 2024-2025 | 给Agent看什么    | 管理上下文、RAG、工具    | 基于上下文推理   |
| Harness Engineering | 2026至今  | 如何设计环境     | 设计约束、反馈、改进机制 | 在约束内自主执行 |

### 2.2 Prompt Engineering的局限

Prompt Engineering关注单次输入提示的优化。它的核心问题是："如何写出能让模型产生好输出的提示词？"

然而，Prompt Engineering存在明显局限：
- 优化目标：单次推理的输出质量
- 问题句式："给模型看什么"
- 失败模式：单次输出不准确
- 典型实现：提示词模板、Few-shot示例

这类似于道德经所说的"有为"——人类试图通过精细控制每一个输入来获得期望输出。

### 2.3 Context Engineering的扩展

Context Engineering更广义地管理Agent看到的上下文。它的核心问题是："如何确保Agent在正确的时机获得正确粒度的信息？"

Context Engineering包含：
- 渐进式披露：将隐性知识转化为结构化信息，通过层级目录按需引导
- 运行时数据接入：提供日志、指标和环境映射，允许Agent使用LogQL/PromQL等工具动态查询
- 直接感知环境：通过CDP等协议让Agent操作浏览器，实现原生的Bug重现或UI验证
- 工具定义、RAG结果、消息历史、MCP数据、仓库知识库

这已经开始体现"无为"的思想——人类不再控制每一个细节，而是设计信息架构让Agent自主选择需要什么信息。

### 2.4 Harness Engineering的升华

Harness Engineering上升到系统级、全生命周期的工程实践。它的核心问题是："系统应该防止什么、测量什么、控制什么、修复什么？"

用mtrajan的话说："Context Engineering问的是'Agent应该看到什么'；Harness Engineering问的是'系统应该防止什么、测量什么、纠正什么'。"

Harness Engineering包含：
- 架构约束：确定性的强制执行
- 熵管理：持续的质量维护
- 工作流控制：Agent的运行时管理
- 反馈循环：实时验证、测试、观测

这正是"无为而治"的完整体现——人类设计Harness（环境、约束、反馈机制），Agent在Harness的引导下自主运行。

### 2.5 三层概念的层级关系

```
┌─────────────────────────────────────────────────────────────┐
│ Harness Engineering（驾驭工程）                              │
│ Agent/工作流的整体系统设计                                    │
│                                                             │
│ + 架构约束（Linters、结构测试、依赖规则）                      │
│ + 反馈循环（CI/CD集成、可观测性）                             │
│ + 工作流控制（任务拆分、并行化、权限管理）                      │
│ + 改进周期（熵管理、自动清理、文档保鲜）                        │
├─────────────────────────────────────────────────────────────┤
│ Context Engineering（上下文工程）                            │
│ 所有输入给LLM的上下文的设计与管理                              │
│                                                             │
│ + 工具定义、RAG结果、消息历史、输出模式、MCP...                │
├─────────────────────────────────────────────────────────────┤
│ Prompt Engineering（提示词工程）                             │
│ 人类给LLM的指令文本优化                                       │
└─────────────────────────────────────────────────────────────┘
```

### 2.6 本章小结

本章从"无为而治"的哲学思想出发，梳理了AI工程从Prompt Engineering到Context Engineering再到Harness Engineering的演进历程。每一次演进都体现了"无为"的深化——人类从直接控制输出，到设计信息架构，再到设计完整的约束和反馈系统。

"我无为，而民自化；我好静，而民自正；我无事，而民自富；我无欲，而民自朴。"在Harness Engineering中，人类工程师通过设计Harness让Agent自主运行，这正是"无为而治"的现代诠释。

## 第三章 三生万物：Prompt → Context → Harness三层架构

*"道生一，一生二，二生三，三生万物。"（《道德经》第四十二章）*

老子用这句话描述了宇宙生成的过程——从"道"生出统一的整体，从统一生出对立，从对立生出变化，从变化生出万物。在AI工程领域，这一思想体现为Prompt → Context → Harness的三层架构演进——从单一提示词，到上下文管理，再到完整的Harness系统。

### 3.1 道生一：Prompt作为起点

"道生一"——从"道"生出统一的整体。Prompt Engineering就是这个"一"，它是人类与AI交互的最基本形式。

Prompt的核心作用：
- 指令：告诉Agent要做什么
- 上下文：提供完成任务所需的背景信息
- 示例：通过Few-shot学习引导输出格式
- 约束：限制输出的范围或格式

然而，单一的Prompt无法应对复杂的软件开发任务。就像"一"需要生出"二"才能产生变化，Prompt需要与更多元素结合才能形成完整的系统。

### 3.2 一生二：Context的双重维度

"一生二"——从统一生出对立。Context Engineering引入了两个关键维度：

**维度一：静态上下文**
- 仓库知识库：代码结构、架构文档、设计规范
- 工具定义：MCP服务器、API接口、函数签名
- RAG结果：从向量数据库检索的相关文档
- 技能定义：SKILL.md文件，描述特定任务的执行方式

**维度二：动态上下文**
- 消息历史：对话的完整记录
- 运行时数据：日志、指标、链路追踪
- 环境状态：当前工作目录、文件系统状态
- Agent进度：任务完成情况、待办事项

静态上下文提供"知识"，动态上下文提供"状态"。两者的结合让Agent能够"知其雄，守其雌"——既了解整体架构，又掌握当前进展。

### 3.3 二生三：Harness的三大支柱

"二生三"——从对立生出变化。Harness Engineering在Context的基础上，增加了第三个维度：系统级的约束和反馈机制。根据Martin Fowler/Thoughtworks的总结，Harness由三大支柱构成：

| 支柱       | 核心问题                          | 关键机制                                 | 道德经对应         |
| ---------- | --------------------------------- | ---------------------------------------- | ------------------ |
| 上下文工程 | Agent在正确时机获得正确粒度的信息 | 渐进式披露、运行时数据接入、直接感知环境 | 知人者智，自知者明 |
| 架构约束   | 将代码品味编码为可强制执行规则    | AI友好型Linter、结构测试、双轨审计机制   | 知其雄，守其雌     |
| 熵管理     | 防止系统随时间腐化                | 反熵Agent、实时演进、持续清理            | 反者道之动         |

### 3.4 三生万物：完整的Harness生态系统

"三生万物"——从变化生出万物。完整的Harness生态系统包含众多组件，共同支撑Agent的可靠运行：

```
┌─────────────────────────────────────────────────────────────────┐
│ Harness生态系统                                                   │
├─────────────────────────────────────────────────────────────────┤
│ 上下文层                                                         │
│ ├── AGENTS.md / CLAUDE.md (渐进式披露)                          │
│ ├── docs/ARCHITECTURE.md (架构文档)                             │
│ ├── SKILL.md (技能定义)                                          │
│ ├── MCP Servers (工具接口)                                       │
│ └── RAG / Vector DB (知识检索)                                  │
├─────────────────────────────────────────────────────────────────┤
│ 约束层                                                           │
│ ├── Linters (代码规范检查)                                       │
│ ├── Structural Tests (架构边界测试)                             │
│ ├── Pre-commit Hooks (提交前检查)                               │
│ └── CI/CD Gates (质量门禁)                                      │
├─────────────────────────────────────────────────────────────────┤
│ 反馈层                                                           │
│ ├── Test Suites (测试套件)                                      │
│ ├── Observability (可观测性)                                    │
│ │   ├── Logs (日志)                                             │
│ │   ├── Metrics (指标)                                          │
│ │   └── Traces (链路追踪)                                       │
│ └── Eval Harness (评估框架)                                     │
├─────────────────────────────────────────────────────────────────┤
│ 进化层                                                           │
│ ├── Garbage Collection Agents (垃圾回收Agent)                   │
│ ├── Documentation Consistency (文档一致性检查)                   │
│ └── Pattern Enforcement (模式执行)                              │
└─────────────────────────────────────────────────────────────────┘
```

### 3.5 本章小结

本章从"三生万物"的哲学思想出发，阐述了Prompt → Context → Harness的三层架构演进。Prompt是起点，Context增加了信息的深度和广度，Harness则构建了完整的约束和反馈系统。三大支柱——上下文工程、架构约束、熵管理——共同支撑起Agent的可靠运行。

"万物负阴而抱阳，冲气以为和。"Harness生态系统中的各个组件，就像阴阳相互作用一样，通过约束与自由、反馈与执行的动态平衡，让Agent能够在正确的道路上持续前进。

## 第四章 上善若水：上下文工程

*"上善若水。水善利万物而不争，处众人之所恶，故几于道。"（《道德经》第八章）*

水，是道德经中最具智慧的隐喻。水无常形，随器而变；水无常势，顺势而为。在Harness Engineering中，上下文工程正是这种"柔性"的体现——不是强制Agent接受固定的信息，而是设计信息架构让Agent按需获取。

### 4.1 上下文工程的核心思想

上下文工程的核心问题是："如何确保Agent在正确的时机获得正确粒度的信息？"

这与传统Prompt Engineering有本质区别：

| 维度     | Prompt Engineering   | Context Engineering       |
| -------- | -------------------- | ------------------------- |
| 优化目标 | 单次推理的输入质量   | 整个系统的持续信息质量    |
| 问题句式 | "给模型看什么"       | "Agent在何时需要什么信息" |
| 失败模式 | 单次输出不准确       | 信息过载或信息不足        |
| 典型实现 | 提示词模板、Few-shot | 渐进式披露、RAG、Memory   |

### 4.2 渐进式披露

渐进式披露是上下文工程的核心模式。它的思想是：将隐性知识转化为结构化信息，通过层级目录按需引导，而非盲目堆砌。

OpenAI团队在实践中发现，单一的巨型AGENTS.md方案失败了。他们转而使用结构化的渐进披露模式：

```
AGENTS.md          ← 约100行的"目录"，指向更深层文档
ARCHITECTURE.md    ← 架构概述
docs/
├── design-docs/   ← 设计文档
├── exec-plans/    ← 执行计划
├── product-specs/ ← 产品规格
├── references/    ← 参考资料
├── DESIGN.md      ← 设计细节
├── FRONTEND.md    ← 前端规范
├── RELIABILITY.md ← 可靠性规范
└── SECURITY.md    ← 安全规范
```

这种结构的优势：
- Agent首先阅读AGENTS.md获取整体概览
- 根据当前任务，按需深入特定文档
- 避免一次性加载过多无关信息
- 类似于人类工程师的工作方式

### 4.3 运行时数据接入

上下文工程不仅包括静态文档，还包括动态的运行时数据。OpenAI团队将Chrome DevTools Protocol集成到Agent运行时，使Codex能：
- 捕获DOM快照和截图
- 暴露日志、指标、链路追踪
- 通过LogQL/PromQL/TraceQL查询
- 让"确保服务启动在800ms内完成"这样的指令成为可验证的目标

这使得Agent能够"自知"——了解自己的运行状态，并据此调整行为。

### 4.4 MCP：Model Context Protocol

MCP（Model Context Protocol）是Anthropic提出的开放标准，定义了应用程序如何与大型语言模型共享上下文。MCP提供了一种标准化的方式，将AI模型连接到不同的数据源和工具。

MCP的核心价值：
- 标准化接口：统一的工具调用协议
- 生态互通：不同Agent可以共享相同的工具
- 安全边界：受控的代码执行环境

截至2026年4月，MCP已有超过10,000个活跃服务器、9700万月下载量，500+社区MCP服务器。涵盖：
- 开发工具：文件编辑、Shell执行、Git操作
- 知识管理：RAG、Memory、知识图谱
- 外部服务：GitHub、Slack、Notion、数据库
- 浏览器自动化：Playwright、Puppeteer

### 4.5 本章小结

本章从"上善若水"的哲学思想出发，深入探讨了上下文工程的核心概念。渐进式披露、运行时数据接入、MCP协议，这些都是让Agent能够"随器而变"、"顺势而为"的技术实践。

"天下莫柔弱于水，而攻坚强者莫之能胜。"在AI技术快速迭代的今天，这种"柔弱"的上下文设计智慧，正是构建持久系统的关键。

## 第五章 知其雄守其雌：架构约束

*"知其雄，守其雌，为天下溪。为天下溪，常德不离，复归于婴儿。"（《道德经》第二十八章）*

这句话的意思是：了解刚强的一面，却保持柔弱的姿态，如同天下的溪谷。作为天下的溪谷，永恒的德就不会离去，回归到婴儿般的纯真状态。在Harness Engineering中，这意味着：了解Agent的强大能力（雄），却用约束来守护它的边界（雌），让它在正确的道路上运行。

### 5.1 架构约束的核心思想

架构约束的核心问题是："如何将代码品味编码为可强制执行的规则？"

OpenAI团队深刻认识到："写在文档里不够——如果不能机械化执行，Agent就会偏离。"他们将"黄金原则"直接编码到仓库中，用自定义Linters和结构测试自动检测违规。

架构约束的目标：
- 定义清晰的边界：什么可以做，什么不可以做
- 机械化执行：让检查自动化，而非依赖人工审查
- 持续验证：每次提交都进行检查
- 快速反馈：发现问题立即修复

### 5.2 OpenAI的黄金原则

OpenAI团队定义了"黄金原则"——带有主观意见的机械规则，保持代码库的可读性和一致性。例如：

**原则1：优先使用共享的utility包，而非手工编写的helper函数**
- 目的：将不变式集中管理
- 实现：自定义Linter检测重复代码
- 自动修复：提供重构建议

**原则2：不使用"YOLO式"数据探测**
- 要求：验证边界，或依赖类型化的SDK
- 目的：防止Agent基于猜测的结构进行构建
- 实现：类型检查、接口验证

### 5.3 依赖方向约束

OpenAI团队定义了每个业务领域的依赖方向：

```
依赖方向（从高到低）：
Types → Config → Repo → Service → Runtime → UI

规则：
- 上层可以依赖下层
- 下层不能依赖上层
- 同层之间可以相互依赖

示例违规：
❌ Service层直接导入UI层的组件
❌ Runtime层直接操作Config层
```

这种依赖方向的约束，确保了代码库的分层清晰，防止了循环依赖和架构腐化。

### 5.4 AI友好型Linter

传统的Linter消息是给人看的，Harness时代的Linter消息直接包含修复建议，便于Agent实现自主闭环。

| 传统Linter              | AI友好型Linter                                               |
| ----------------------- | ------------------------------------------------------------ |
| 'Unused variable foo'   | 'Remove unused variable foo on line 42. Suggested fix: delete line 42.' |
| 'Import cycle detected' | 'Import cycle detected: A→B→C→A. Break cycle by extracting common code to new module D.' |
| 'Function too long'     | 'Function processData (lines 10-150) exceeds 50 lines. Consider splitting into: validateInput, transformData, saveResult.' |

```python
# Python示例：使用ArchUnit风格的结构测试
import ast
import os

def test_dependency_direction():
    """验证依赖方向约束"""
    violations = []
    
    for root, dirs, files in os.walk('src'):
        for file in files:
            if file.endswith('.py'):
                filepath = os.path.join(root, file)
                layer = get_layer(filepath)  # Types/Config/Repo/Service/Runtime/UI

                            })
    
    assert len(violations) == 0, f"Dependency violations: {violations}"
```

### 5.6 本章小结

本章从"知其雄，守其雌"的哲学思想出发，深入探讨了架构约束的核心概念。黄金原则、依赖方向约束、AI友好型Linter、结构测试，这些都是"守其雌"的技术实践——用约束来守护Agent的边界，让它在正确的道路上运行。

"常德不离，复归于婴儿。"当架构约束成为代码库的本能，Agent就能像婴儿一样，在正确的环境中自然而然地成长。

## 第六章 反者道之动：熵管理

*"反者道之动，弱者道之用。天下万物生于有，有生于无。"（《道德经》第四十章）*

"反"是道的运动规律——事物发展到极致就会向相反方向转化。在Harness Engineering中，这一思想体现为熵管理：当代码库积累足够的"混乱"时，必须有机制将其"反转"回有序状态。

### 6.1 什么是熵增

AI生成的代码库会随着时间积累"熵"——混乱：
- 文档与代码脱节
- 命名规范漂移
- 死代码堆积
- 架构约束被悄悄打破
- 重复代码蔓延

OpenAI团队最初靠人工解决这个问题：每周五，整个团队（20%的工时）用于手动清理"AI垃圾代码"。这显然不可持续。

### 6.2 熵管理的核心思想

熵管理的核心问题是："如何让Agent来清洁Agent留下的乱摊子？"

解决方案：建立一组后台"垃圾回收Agent"，周期性运行：

| 垃圾回收Agent类型 | 任务                                 | 执行频率 |
| ----------------- | ------------------------------------ | -------- |
| 文档一致性Agent   | 验证文档与当前代码是否匹配，开修复PR | 每日     |
| 约束违规扫描器    | 发现绕过了早期检查的约束违反         | 每PR     |
| 模式执行Agent     | 识别并修复偏离既定模式的代码         | 每周     |
| 依赖审计器        | 追踪并解决循环或不必要的依赖         | 每周     |

### 6.3 垃圾回收机制

OpenAI团队的垃圾回收机制：
1. 编码黄金原则到仓库中
2. 建立定期运行的后台Codex任务
3. 自动化合并（大部分清理PR可在一分钟内审阅）

这类似于垃圾回收（GC）的持续技术债清理。技术债务就像一笔高息贷款：不断地以小额贷款的方式偿还，总比让债务不断累积再痛苦地一次解决要好得多。

### 6.4 熵管理的数学模型

```
熵增模型：
E(t) = E₀ + α·t - β·∫C(τ)dτ

其中：
- E(t): 时间t时的系统熵值
- E₀: 初始熵值
- α: 自然熵增速率（Agent生成代码引入的混乱）
- β: 熵管理效率系数
- C(τ): 时间τ时的清理强度

关键洞察：
当 β·C(t) > α 时，系统熵值下降（有序度提高）
当 β·C(t) = α 时，系统熵值稳定（动态平衡）
当 β·C(t) < α 时，系统熵值上升（趋向混乱）
```

### 6.5 熵管理的最佳实践

基于OpenAI和Anthropic的实践，熵管理的最佳实践包括：
1. **尽早开始**：不要等到代码库变得混乱才开始
2. **自动化优先**：人工清理不可持续，将清理规则编码为可自动执行的Agent
3. **小步快跑**：频繁的微小清理优于偶尔的大清理
4. **度量驱动**：建立熵值度量指标，追踪清理效果

### 6.6 本章小结

本章从"反者道之动"的哲学思想出发，深入探讨了熵管理的核心概念。熵增是自然规律，但Harness Engineering通过垃圾回收机制，实现了"反"的运动——将混乱的系统重新带回有序状态。

"为学日益，为道日损。损之又损，以至于无为。"在熵管理的道路上，我们也在不断学习"减少"的智慧——减少混乱、减少技术债务、减少人工干预。

# 第三部分：Harness核心机制

## 第七章 大器晚成：反馈循环设计

*"大方无隅，大器晚成，大音希声，大象无形。"（《道德经》第四十一章）*

伟大的器物需要时间来成就。在Harness Engineering中，反馈循环的设计不是一蹴而就的，而是需要持续迭代和完善。一个优秀的反馈循环，就像"大器"一样，需要经历时间的打磨才能成熟。

### 7.1 反馈循环的核心思想

反馈循环的核心问题是："如何让Agent知道自己的行为是否正确，并据此调整？"

OpenAI团队强调："将可观测性接入Agent。"这意味着Agent不仅能生成代码，还能：
- 运行测试并查看结果
- 查询日志了解系统状态
- 获取性能指标
- 根据反馈调整下一步行动

### 7.2 反馈循环的组成

一个完整的反馈循环包含四个环节：

```
┌─────────────┐
│  执行行动   │
│ (Agent执行) │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  收集反馈   │
│ (测试/日志/ │
│ 指标/追踪)  │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  分析结果   │
│ (成功/失败/ │
│ 部分成功)   │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  调整策略   │
│ (继续/重试/ │
│ 改变方法)   │
└──────┬──────┘
       │
       └──────────→ 回到"执行行动"
```

### 7.3 测试作为反馈

测试是最直接的反馈机制。Harness应该为Agent提供：
- 单元测试：验证单个函数/组件的正确性
- 集成测试：验证多个组件的协作
- 端到端测试：验证完整用户流程
- 架构测试：验证架构约束是否被遵守

Agent应该能够在每次代码修改后自动运行测试，并根据结果调整：
1. Agent生成代码
2. 自动运行相关测试
3. 如果测试通过 → 继续下一步
4. 如果测试失败 → 分析错误信息 → 修复代码 → 回到步骤2
5. 如果无法修复 → 请求人类帮助

### 7.4 可观测性作为反馈

| 可观测性类型         | 反馈内容                 | Agent使用方式           |
| -------------------- | ------------------------ | ----------------------- |
| Logs（日志）         | 应用运行时的详细记录     | 通过LogQL查询特定错误   |
| Metrics（指标）      | 性能数据（延迟、吞吐量） | 验证"服务启动在800ms内" |
| Traces（链路追踪）   | 请求在系统中的流转路径   | 定位性能瓶颈            |
| Profiles（性能分析） | CPU/内存使用情况         | 发现资源泄漏            |

### 7.5 反馈循环的设计原则

设计有效的反馈循环需要遵循以下原则：
1. **快速反馈**：反馈应该尽快到达Agent，延迟会降低学习效率
2. **明确反馈**：反馈信息应该清晰明确，错误信息应包含修复建议
3. **可操作反馈**：Agent应该能够根据反馈采取行动，避免模糊反馈
4. **分级反馈**：不同严重程度的反馈应有不同响应，严重错误立即停止

### 7.6 本章小结

本章从"大器晚成"的哲学思想出发，深入探讨了反馈循环的设计。优秀的反馈循环需要时间来打磨，但一旦建立，就能让Agent在正确的道路上持续进化。

"大音希声，大象无形。"最好的反馈循环，是让Agent几乎感受不到它的存在，却能自然而然地根据反馈调整行为。

## 第八章 知足不辱：Agent评估与度量

*"知足不辱，知止不殆，可以长久。"（《道德经》第四十四章）*

知道满足就不会受到屈辱，知道适可而止就不会遇到危险。在Agent评估中，这意味着：知道Agent的能力边界，在合适的场景使用合适的评估方法，才能让Harness系统长久稳定运行。

### 8.1 Agent评估的挑战

评估Agent比评估传统软件更加困难：
- 非确定性：同样的输入可能产生不同的输出
- 长轨迹：Agent的执行可能包含数十个步骤
- 主观性："好"的定义可能因人而异
- 环境依赖：结果可能依赖于执行环境

### 8.2 评估方法论

Agent评估主要有三种方法：

| 评估方法       | 适用场景             | 优点           | 缺点               |
| -------------- | -------------------- | -------------- | ------------------ |
| 基于规则的评估 | 有明确正确答案的任务 | 客观、可重复   | 无法处理开放式任务 |
| 基于模型的评估 | 需要主观判断的任务   | 可处理复杂场景 | 可能引入模型偏见   |
| 人工评估       | 关键决策、最终验收   | 最可靠         | 成本高、速度慢     |

### 8.3 pass@k和pass^k指标

pass@k是评估Agent可靠性的重要指标。它表示：在k次尝试中，至少有一次成功的概率。

```
pass@k 计算公式：
pass@k = 1 - C(n - c, k) / C(n, k)

其中：
- n: 总尝试次数
- c: 成功次数
- k: 评估的尝试次数
- C: 组合数

示例：
- 100次尝试中成功了80次
- pass@1 = 80/100 = 80%
- pass@5 = 1 - C(20,5)/C(100,5) ≈ 99.9%

pass^k（pass to the k）:
表示连续k次都成功的概率
pass^k = (pass@1)^k
```

### 8.4 LangChain六维评估体系

LangChain团队在2026年3月提出，不应盲目堆砌评估指标。每个评估都是一个向量，会改变Agent系统的行为。他们定义了六个核心维度：

1. **文件操作**：读取、写入、移动、删除文件的正确性
2. **检索**：从向量数据库或知识库中检索相关文档的准确率
3. **工具使用**：调用外部API或函数的参数正确性
4. **记忆**：跨会话信息保留和回忆的完整性
5. **对话处理**：多轮对话中的上下文理解和一致性
6. **摘要生成**：长文本摘要的忠实度和简洁度

### 8.5 微软Evals for Agent Interop

2026年2月，微软开源了Evals for Agent Interop，提供针对邮件、日历、文档、协作工具等场景的标准化评估数据集和框架。支持在本地或云端运行，输出JSON格式的详细报告。

### 8.6 One-Eval自动化评估系统

2026年3月，arXiv论文2603.09821提出了One-Eval系统，将自然语言的评估请求（如"检查Agent是否在发送邮件前询问用户确认"）转化为可执行的评估工作流。包含：
- NL2Bench：将意图结构化
- BenchResolve：解析并标准化数据集
- Metrics & Reporting：自动生成评估报告
- 人类介入检查点

### 8.7 Eval-Driven Development (EDD)

Eval-Driven Development是一种新的开发范式，类似于TDD，但针对AI Agent：
1. 先定义评估：明确"成功"的标准
2. 再实现功能：让Agent完成任务
3. 运行评估：验证Agent是否达到标准
4. 迭代改进：根据评估结果优化Agent

EDD的好处：
- 明确的完成标准
- 可量化的改进
- 防止回归

### 8.8 本章小结

本章从"知足不辱"的哲学思想出发，探讨了Agent评估与度量的方法。知道Agent的能力边界，选择合适的评估方法，才能让Harness系统长久稳定运行。

"知止不殆。"知道什么时候停止，什么时候需要人类介入，这是设计Harness评估系统的重要智慧。

## 第九章 治大国若烹小鲜：长周期Agent的状态管理

*"治大国，若烹小鲜。以道莅天下，其鬼不神。"（《道德经》第六十章）*

治理大国就像煎小鱼，不能频繁翻动，否则鱼就碎了。在长周期Agent的状态管理中，这意味着：不能频繁地重置Agent状态，否则任务就无法完成。需要设计优雅的状态管理机制，让Agent能够在多次会话中持续工作。

### 9.1 长周期Agent的挑战

AI Agent必须在离散会话中工作，每个新会话从零开始，没有前一次的记忆。就像一个软件项目由轮班的工程师负责，每个新工程师到岗时对上一班发生了什么毫无记忆。

主要挑战：
- 上下文丢失：每次新会话都从零开始
- 进度追踪：如何知道任务完成了多少
- 状态恢复：如何从上次停止的地方继续
- 一致性：如何保证多次会话的一致性

### 9.2 Anthropic的两阶段解决方案

Anthropic在2025年11月发表了"Effective harnesses for long-running agents"，提出了两阶段解决方案：

| 阶段   | 角色              | 关键行为                                                     |
| ------ | ----------------- | ------------------------------------------------------------ |
| 初始化 | Initializer Agent | 首次运行时搭建环境。创建init.sh、claude-progress.txt、功能清单（JSON）、初始Git提交 |
| 编码   | Coding Agent      | 每个后续会话：读取进度文件和Git日志 → 选择一个功能 → 增量推进 → 提交并更新进度 |

### 9.3 Claude Code "Tasks" 更新的工程突破

2026年1月，Claude Code的"Tasks"更新将"待办清单"从对话内存中的轻量级清单升级为文件系统中的持久化"任务"。核心设计：

- **DAG依赖图**：任务可以明确"阻塞"另一个，防止"幻觉完成"错误
- **文件系统持久化**：任务存储在`~/.claude/tasks/`，支持跨会话、跨机器恢复
- **环境变量编排**：通过`CLAUDE_CODE_TASK_LIST_ID`，多个Claude实例可以指向同一个任务列表

### 9.4 状态管理的关键组件

长周期Agent的状态管理需要以下组件：

```

   - 格式：JSON
```

## 第十章 祸兮福之所倚：故障恢复与容错设计

*"祸兮福之所倚，福兮祸之所伏。孰知其极？其无正也。"（《道德经》第五十八章）*

灾祸中倚藏着福分，福分中潜伏着灾祸。在Harness Engineering中，故障不是终点，而是改进的契机。设计良好的故障恢复机制，能够将"祸"转化为"福"，让Agent系统更加健壮。

### 10.1 故障不可避免

Agent系统会遇到各种故障：
- API错误：LLM API超时或返回错误
- 工具失败：MCP服务器不可用
- 代码错误：Agent生成的代码有Bug
- 环境问题：测试环境配置错误
- 资源耗尽：Token限制、内存不足

### 10.2 故障恢复策略

Harness应该提供多层次的故障恢复策略：

| 策略                    | 适用场景     | 实现方式                   |
| ----------------------- | ------------ | -------------------------- |
| 重试（Retry）           | 临时性错误   | 指数退避重试               |
| 降级（Degradation）     | 主路径失败   | 切换到备用方案             |
| 熔断（Circuit Breaker） | 持续失败     | 暂时停止调用，防止级联故障 |
| 人工介入                | 无法自动恢复 | 通知人类工程师             |

### 10.3 指数退避重试

```
指数退避公式：
delay = min(base * 2^attempt + jitter, max_delay)

其中：
- base: 基础延迟（如1秒）
- attempt: 重试次数
- jitter: 随机扰动（防止惊群效应）
- max_delay: 最大延迟上限

示例：
第1次重试：1秒
第2次重试：2秒
第3次重试：4秒
第4次重试：8秒
```

### 10.4 熔断器模式

熔断器模式防止级联故障：

```
熔断器状态：

┌──────────┐ 错误率超过阈值 ┌──────────┐
│  关闭    │ ───────────→ │  打开    │
│ (Closed) │               │ (Open)   │
│ 正常请求 │ ←─────────── │ 拒绝请求 │
└──────────┘ 超时后尝试恢复 └──────────┘
     ↑                │
     └────────┐       │
              │       │
              │       ▼
          ┌──────────┐
          │  半开    │
          │(Half-Open)│
          │ 试探请求 │
          └──────────┘
```

### 10.5 本章小结

本章从"祸兮福之所倚"的辩证思想出发，探讨了故障恢复与容错设计。故障不是坏事，关键在于如何应对。设计良好的故障恢复机制，能够将危机转化为改进的契机。

"孰知其极？其无正也。"没有人知道故障和成功的边界在哪里。Harness工程师的任务，就是设计机制让Agent能够在故障中学习、恢复、进化。

### 11.3 核心机制：任务调度算法

OpenClaw使用基于优先级的DAG来管理任务：

```python
class TaskScheduler:
    def schedule(self, task_graph):
        for task in task_graph.nodes:
            task.priority = self.calc_priority(task)
        sorted_tasks = topological_sort(task_graph)
        sorted_tasks.sort(key=lambda t: t.priority, reverse=True)
        running = []
        for task in sorted_tasks:
            if not self.conflicts(task, running):
                self.execute_async(task)
                running.append(task)
            if len(running) >= 3:
                self.wait_any(running)
```

创新点：**"推测执行"**——当Agent不确定下一步时，同时尝试2-3个可能的行动路径。

# 第五部分：Harness工程实践

## 第十八章 Harness的十字路口：做厚还是做薄？

*"知其雄，守其雌，为天下溪。"（《道德经》第二十八章）*

2026年3月，Anthropic和OpenAI几乎同时发布了自己的Harness设计哲学，却指向了截然相反的方向。

### 18.1 Anthropic的立场：做强、做厚

Anthropic认为Agent模型虽然强大，但本质上是不确定的、容易偏离轨道的。因此，Harness必须足够"厚"：
- 结构化交接：强制Agent在上下文耗尽前生成JSON进度文件
- 多智能体协作：Planner、Generator、Evaluator分工
- 上下文重置机制：定期清空对话历史，仅保留结构化产物

典型Harness配置：20+工具，15+强制检查点。

### 18.2 OpenAI Codex的立场：做小、做薄

OpenAI Codex开源负责人Michael Bolin明确提出："Harness不应该无限膨胀。理想状态下，Harness应该尽可能小，模型应该尽可能强。"

Codex的设计理念：
- 减少工具数量：从40+削减到12个核心工具
- 避免过度干预：移除大部分检查点
- 信任模型：相信模型能够从错误中学习

### 18.3 分歧的本质

| 维度         | Anthropic（厚）          | OpenAI Codex（薄）         |
| ------------ | ------------------------ | -------------------------- |
| 对模型的假设 | 模型容易犯错，需要强约束 | 模型足够聪明，可以自我纠正 |
| Harness职责  | 防止所有已知错误         | 提供基础能力，允许犯错     |
| 迭代方式     | 发现错误→增加约束        | 发现错误→改进模型          |
| 适用场景     | 生产环境、高风险领域     | 探索性任务、原型开发       |

**分歧的本质**：对"模型能力演进速度"的不同判断。

### 18.4 第三条路：分层Harness

DeerFlow 2.0提出分层Harness策略：

```yaml
layers:
  core:        # 永不妥协的薄层
    - sandbox
    - permission_gate
    - audit_log
  adaptive:    # 可调节的厚层
    - tool_whitelist
    - step_approval
  optional:    # 完全可选的厚层
    - planner_agent
    - reviewer_agent
```

### 18.5 本章小结

无论厚薄，以下原则通用：
1. 从薄开始，按需加厚
2. 区分"永久约束"与"临时约束"
3. 为Harness的"瘦身"留出空间

> *"为学日益，为道日损。"* 构建Harness既是增加约束，也是简化约束。

## 第二十一章 为学日益为道日损：约束与自由度平衡

*"为学日益，为道日损。损之又损，以至于无为。"（《道德经》第四十八章）*

在Harness设计中，这意味着：约束越多，Agent的自由度越少；但恰到好处的约束，反而能让Agent更有效地工作。

### 21.1 约束与自由度的辩证关系

| 约束过多            | 约束过少          |
| ------------------- | ----------------- |
| Agent无法发挥创造力 | Agent容易偏离目标 |
| 解决方案空间过小    | 解决方案空间过大  |
| Agent感到束缚       | Agent容易迷失     |

### 21.2 Mitchell Hashimoto的方法

> *"每当Agent犯一个新类型的错误，就回头加一条约束。"*

日积月累，Harness越来越健壮，Agent能犯的错越来越少——这是一个正向的复利飞轮。

### 21.3 约束设计原则

| 原则     | 说明                              |
| -------- | --------------------------------- |
| 必要性   | 只添加必要的约束                  |
| 可执行性 | 约束必须能被机械化执行            |
| 清晰性   | 约束的描述必须清晰明确            |
| 可恢复性 | 违反约束后，Agent应该知道如何恢复 |

### 21.4 本章小结

"损之又损，以至于无为。"当约束成为代码库的本能，Agent就能"无为而治"，自然而然地产生正确的输出。

## 附录B：Harness Engineering术语表（扩充）

**Harness Engineering**：驾驭工程，系统性地设计约束、反馈循环和持续改进机制，使AI Agent能够在大规模、长周期的软件开发中可靠运行。

**Context Engineering**：上下文工程，管理所有输入给LLM的上下文的设计与管理。

**Prompt Engineering**：提示词工程，优化人类给LLM的指令文本。

**Architectural Constraints**：架构约束，将代码品味编码为可强制执行的规则。

**Entropy Management**：熵管理，防止AI生成的代码库随时间腐化的机制。

**Golden Principles**：黄金原则，代码库的规范DO/DON'T模式。

**Garbage Collection**：垃圾回收，定期运行的Agent清理代码库的机制。

**AGENTS.md**：Agent指南文件，约100行的渐进披露地图。

**MCP**：Model Context Protocol，模型上下文协议，标准化Agent与工具的交互。

**pass@k**：k次尝试中至少一次成功的概率，Agent可靠性指标。

**Eval-Driven Development**：评估驱动开发，先定义评估再实现功能的开发范式。

**AgentOps**：Agent运维，AI Agent的可观测性和运维实践。

**Feedback Loop**：反馈循环，让Agent知道行为是否正确并据此调整的机制。

**Two-Agent Pattern**：双Agent模式，Initializer Agent + Coding Agent的协作模式。

**DAG**：有向无环图，用于任务依赖管理。

**CDP**：Chrome DevTools Protocol，用于浏览器自动化的协议。

**Skill**：技能，可复用的Agent能力单元。

**Sandbox**：沙箱，隔离的代码执行环境。
